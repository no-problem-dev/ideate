# Diversity Engineering Principles

## CONTEXT

### 適用範囲

本ルールは ideate プラットフォーム配下の **すべてのスキル・エージェント** に適用される。
繰り返し実行されるスキルが毎回同じ結論に収束する「モードコラプス」問題に対抗するための設計原則を定義する。

### 用語

| 用語 | 定義 |
|------|------|
| モードコラプス | LLM の出力が繰り返し実行で同じ結論に収束する現象 |
| 質の高い多様性 | 品質閾値を満たす出力間での意味的な多様性 |
| セレンディピティ予算 | 意図的に非効率な探索に割り当てるリソース比率 |
| ローテーションテーブル | 日付ベースで入力パラメータを決定論的に切り替えるテーブル |
| Novelty スコア | 過去出力との差分を定量化した新規性指標 |

### 前提条件

- LLM は RLHF/RLAIF により「典型的な出力」に収束する傾向を持つ
- Temperature/Top-p の調整だけでは質の高い多様性は実現できない
- 過去出力を参照しない限り、セッション間の多様性は構造的に低い

### 根拠となる研究

| 論文 | 主要知見 |
|------|---------|
| G2 (EMNLP 2025) | 過去出力を活用しない限り根本的な多様性問題は解決しない |
| MoA (ICLR 2025) | 異なるモデル/戦略の組み合わせが同一モデル繰り返しより常に優れる |
| DMAD (ICLR 2025) | ペルソナより推論戦略の多様化が効果的 |
| CreativeDC (2025) | 発散-収束分離でベースラインより 72% 高い意味的多様性 |
| SIGDIAL 2025 | 批評側の多様化が提案側より最終成果物の質を向上させる |

## ROLE

| 属性 | 内容 |
|------|------|
| **専門性** | LLM 出力の多様性設計、確率的生成モデルの収束制御 |
| **権限** | 全スキルの多様性戦略の定義、品質-多様性トレードオフの基準設定 |
| **責任** | 繰り返し実行スキルの出力新規性の維持 |
| **禁止事項** | 品質閾値を下回る出力を多様性の名目で許容すること |

## INTENT

### Goal

ideate プラットフォーム配下のスキルが、繰り返し実行においても質の高い多様性を維持するための設計原則を提供する。

### Value

- **V-1**: 同じスキルを週次で回しても、毎回異なる価値ある洞察が得られる
- **V-2**: 多様性の実現方法がスキル横断で一貫している
- **V-3**: 品質と多様性のトレードオフが明示的に管理されている

### Success Criteria

- [ ] 繰り返し実行スキルの Featured 出力の重複率が 20% 以下
- [ ] 各スキルが少なくとも 2 つ以上の原則を実装している
- [ ] Novelty スコアが評価軸に含まれている

## STEPS

### 7 つの設計原則

#### DEP-1: Memory-First（記憶なき多様性は幻想）

**最重要原則。他の全原則の前提条件。**

繰り返し実行するスキルは、実行前に過去 N 回分の出力サマリーをコンテキストに注入する。

**実装**: `core/sequences/diversity-context-loader.md` を参照。

**適用判定**:
- 繰り返し実行 × 探索的タスク → **必須**
- 繰り返し実行 × 記録的タスク → 不要
- 単発実行 → 不要

---

#### DEP-2: Input Diversity（入力を変えろ、出力を変えるな）

出力側のランダム化（Temperature）ではなく、入力側の多様化で対応する。
検索クエリ、情報ソース、分析視点をローテーションテーブルで計画的に変化させる。

**実装**: `core/sequences/rotation-engine.md` を参照。

**ローテーション軸**:
- **視点** (perspective): 経済的 / 感情的 / 行動科学的 / 技術的
- **クエリフレーム** (query_frame): 問題探索 / 解決策探索 / 失敗分析 / アナロジー探索
- **追加ソース** (extra_source): ドメイン固有のソースプール

---

#### DEP-3: Reasoning Strategy Diversity（推論戦略を変えろ）

ペルソナの変更は出力の多様性に限定的な効果しかない。
推論の仕方自体を変える方が本質的に異なる解空間を探索できる。

**推論戦略の例**:
- 直接回答（IO）
- ステップバイステップ推論（CoT）
- 反実仮想推論（「もし〜でなかったら」）
- フレームワーク適用（JTBD / 行動経済学 / システム思考）

**適用方法**: 並列エージェントを立てる際、ペルソナではなく分析フレームワークを変える。

---

#### DEP-4: Critic Diversity（批判は提案より多様であるべき）

提案側の多様化 → 提案の多様性向上
批評側の多様化 → **最終成果物の実現可能性向上**

評価・検証フェーズでは、複数の異なる観点から批判を行う。

**実装**: `core/sequences/parallel-critique.md` を参照。

---

#### DEP-5: Diverge-Converge Separation（発散と収束を分離せよ）

発散フェーズでは制約を外し、収束フェーズで制約を戻す。
一度に両方やろうとすると、LLM は安全な中間地帯に落ち着く。

**実装**: `core/sequences/diverge-converge.md` を参照。

---

#### DEP-6: Serendipity Budget（セレンディピティ予算を確保せよ）

全探索の 20-30% を「意図的に非効率な探索」に割り当てる。
クロスドメインアナロジー、ランダムソース探索、低スコアカテゴリの強制探索など。

**適用方法**: スキル定義内でセレンディピティ枠を明示的に確保する。
例: 7 件のアイデア中 1-2 件はアナロジー探索から強制生成。

---

#### DEP-7: Novelty as First-Class Metric（新規性を正式な評価軸にせよ）

スコアリングに「新規性（N）」を追加する。
過去出力との意味的類似度から算出し、繰り返し登場するアイデアにペナルティを与える。

**スコア定義**:
```
N（Novelty）: 1-5
  5 = 過去 6 ヶ月に類似出力なし
  4 = 過去 3 ヶ月に類似出力なし
  3 = 類似はあるが切り口が異なる
  2 = 類似出力が過去 1 ヶ月以内にあり
  1 = ほぼ同一の出力が直近に存在
```

**前提**: DEP-1（Memory-First）が実装されていること。

---

### 適用マトリクス

| スキル特性 | 必須原則 | 推奨原則 |
|-----------|---------|---------|
| 繰り返し × 探索的 | DEP-1, DEP-7 | DEP-2〜6 全て |
| 繰り返し × 記録的 | なし | なし |
| 単発 × 探索的 | DEP-5 | DEP-3, DEP-4 |
| 単発 × 記録的 | なし | なし |

## PROOF

### Verification

- [ ] 繰り返し実行スキルが DEP-1（Memory-First）を実装しているか
- [ ] ローテーションテーブルが定義され、実際に異なる入力を生成しているか
- [ ] Novelty スコアが評価軸に含まれ、過去出力との差分が反映されているか
- [ ] セレンディピティ枠が明示的に確保されているか

### Done Criteria

- [ ] 全ての繰り返し実行スキルが少なくとも DEP-1 + DEP-7 を実装
- [ ] 4 週間のテスト運用で Featured 出力の重複率 20% 以下を達成

### Do / Don't

**Do**:
- ✅ 多様性の仕組みをスキル定義に明示的に記述する
- ✅ 品質閾値を維持した上で多様性を追求する
- ✅ 過去出力を必ず参照してから新規生成する

**Don't**:
- ❌ Temperature を上げるだけで多様性を確保したとみなす
- ❌ 品質を犠牲にして多様性を追求する
- ❌ ペルソナ変更だけで多様性を実現したとみなす
